        # SSEStore
        Global store for handling SSE events from one or more connections, built with Zustand. The store manages named `SSEConnection` instances, emits and keeps a snapshot of the latest unified message, and exposes both reactive hooks and imperative helpers to read messages and manage connections.
        
        The Slice SSE store has a narrow goal: provide of a singleton global state for the last message received and emit that as state. Storing, persisting, accumulating and manipulating SSE message events is logic is left to the consuming app.
            
        ## Adding a new SSE connection
        `addConnection` creates a new `SSEConnection` and saves it to state. If a connection with the same name already exists, the store closes the old connection before replacing it. The store always creates connections in unified mode, so the options you pass here are the `SSEConnection` options except for `unifiedOnMessage`, which the store manages internally.
        A typical example might look like this:

	        <RenderSource language={'tsx'}>
	        ```
	        import { useEffect, useRef } from 'react';
	        import { useSSE, useIsConnected, useMessage, SSEStoreConnection } from '@apple-pie/slice/stores/SSE';
	        
	        // your Api's custom event types, if it emits any
	        type Progress = number;
            type UserCreated = { id: string; email: string };
	            
	        function AppActivityStream() {
                // you can use a ref to avoid (re)adding the same connection on re-renders
                const connectionRef = useRef<SSEStoreConnection | null>(null);
                // use store hooks to add/remove connections and read event messages
	            const { addConnection, removeConnection } = useSSE();
	            const connected = useIsConnected('activity');
	            const anyMessage = useMessage();
	            const progressMessage = useMessage<Progress>('progress');
	            const userCreatedMessage = useMessage<UserCreated>('userCreated');
	                
	            // add the new SEE connection
	            useEffect(() => {
                    if(connectionRef.current) return
                    connectionRef.current = addConnection('activity', {
                        url: '/api/events',
                        options: { withCredentials: true },
                        customEvents: [
                            { name: 'progress' },
                            { name: 'userCreated' },
                        ],
	                });
                    // place the connection clean up where it makes sense for your app
	                return () => {
                        removeConnection('activity');
                        connectionRef.current = null;
                    }
	            },[addConnection, removeConnection]);
	            
	            return (
	                <div>
	                    connected: {connected ? 'yes' : 'no'} -
	                    progress: {progressMessage ?? 'waiting...'} - 
	                    userCreated: {userCreatedMessage?.email ?? 'waiting...'} - 
	                    last type: {anyMessage?.type ?? 'waiting...'}
	                </div>
	            );
	        }
	        ```
	        </RenderSource>
	        
	        Note: Slice stores are singletons, which means you can safely import `useMessage` anywhere in your app and subscribe to the latest matching SSE update.
            If you plan to make events available globally to your app, make sure to place your connection cleanup where it makes sense.

        ## Removing a connection
        Removing a connection always closes it first, then removes it from the store.

        Note: `SSEConnection` also supports a self-cleanup path through its optional `connectionClose` setting. When that configured close condition is matched, the connection emits a synthetic `close` event and the store automatically removes it. Even with that built-in behavior, it is still good practice to explicitly close/remove connections in your own cleanup paths as well, since network errors, stalled streams, component unmounts, or unexpected backend behavior may prevent the normal self-close signal from ever arriving.

        <RenderSource language={'tsx'}>
            {`
import { useSSEActions } from '@apple-pie/slice/stores/SSE';

useSSEActions.removeConnection('activity');
`}
        </RenderSource>

import { useSSEActions, useLastSSEMessage } from '@apple-pie/slice/stores/SSE';

useSSEActions.addConnection('system', {
  url: '/api/system/events',
  customEvents: [{ name: 'progress' }],
});

export function getLatestSSEType() {
  return useLastSSEMessage()?.type ?? null;
}
