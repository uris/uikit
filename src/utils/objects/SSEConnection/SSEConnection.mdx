        # SSEConnection Class
        Typed wrapper around the browser `EventSource` API. It supports:
        - the default `message` event
        - built-in `open`, `error`, and synthetic `close` lifecycle events
        - custom named SSE events with event-specific payload types
        - either standard message callbacks or unified callback mode

        ## Import
        <RenderSource language={'tsx'}>
{`
import { SSEConnection } from '@apple-pie/slice/utils/objects';
`}
</RenderSource>

        ## Self-closing connection
        `connectionClose` lets the SSE connection close itself automatically when one of two configured conditions is met:
        <table style={{width: '100%'}}>
          <colgroup>
            <col style={{width: '1%'}} />
            <col style={{width: '25%'}} />
            <col style={{width: '98%'}} />
          </colgroup>
          <thead>
            <tr>
              <th className={'left'}>Field</th>
              <th className={'left'}>Type</th>
              <th className={'left'}>Description</th>
            </tr>
          </thead>
          <tbody>
            <tr><td><code>event</code></td><td><code>string</code></td><td>Closes the connection when a matching custom SSE event name is received.</td></tr>
            <tr><td><code>message</code></td><td><code>string</code></td><td>Closes the connection when the default SSE <code>message</code> payload matches that string.</td></tr>
          </tbody>
        </table>

        When a configured self-close trigger fires, `SSEConnection` emits a synthetic built-in `close` event through unified mode and calls `onCloseCallback`.

        <RenderSource language={'tsx'}>
{`
const connection = new SSEConnection({
	url: '/api/events',
	unifiedOnMessage: true,
	connectionClose: { message: 'eos' },
	onCloseCallback: () => {
		console.log('Stream closed itself');
	},
	onMessageCallback: (message) => {
		if (message.type === 'close') {
			console.log(message.event);
		}
	},
});
`}
</RenderSource>

        ## "Standard" mode
        Standard mode is the default. Only the default `message` event is routed through `onMessageCallback` - in it's simplest form, just create the new connection and pass it a callback handler for message events. 

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection({
	url: '/api/events',
	onMessageCallback: (data) => {
		console.log(data);
	},
});
```
</RenderSource>

        Optionally, add the `open`, `error`, and self-close `close` callbacks if you need to handle these events as well...
        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection({
	url: '/api/events',
	onMessageCallback: (data) => console.log(data),
	onOpenCallback: (event) => console.log(event),
	onErrorCallback: (event) => console.log(event),
	onCloseCallback: (event) => console.log(event),
});
```
</RenderSource>
            
        ## "Unified mode"
        Unified mode routes all built-in and custom events through a single callback. The `type` field identifies the event type.

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection({
	url: '/api/events',
	unifiedOnMessage: true, // set flag to enable unified mode
	onMessageCallback: (message) => {
		switch (message.type) {
			case 'message':
				console.log(message.data);
				break;
			case 'open':
				console.log(message.event);
				break;
			case 'error':
				console.log(message.event);
				break;
			case 'close':
				console.log(message.event);
				break;
		}
	},
});
```
</RenderSource>

        ## Custom events
        SSE allows defining and emitting custom events from your API/backend. With Slice SSE Connection, you define custom events as an event map. Each key of the map is the SSE event name (the same as emitted by your back end) and each value is the payload type for that event.

        <RenderSource language={'tsx'}>
```
// each key is the event name, and each value, the payload type
type SSECustomEvents = {
	progress: number;
	userCreated: { id: string };
};
```
</RenderSource>

            Then, when creating a new connection, set the custom event type in the connection generics: `new SSEConnection<unknown, SSECustomEvents>`. Also pass in your custom events with a custom callback handler for each event.

        <RenderSource language={'tsx'}>
```
// note: use 'unknown' type for the default message payload type if it's type is not known
const connection = new SSEConnection<unknown, SSECustomEvents>({
	url: '/api/events',
	customEvents: [
		{ name: 'progress', handler: (progress) => console.log(progress) },
		{ name: 'userCreated', handler: (user) => console.log(user.id) },
	],
});
```
</RenderSource>

        NOTE: The `handler` field of custom events is optional. If no handler is defined, custom event messages will still flow through unified callback mode when `unifiedOnMessage` is enabled:

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection<unknown, { progress: number }>({
	url: '/api/events',
	unifiedOnMessage: true, // set flag to enable unified mode
	customEvents: [{ name: 'progress' }],
	onMessageCallback: (message) => {
		if (message.type === 'progress') {
			console.log(message.data);
		}
	},
});
```
</RenderSource>

        ## Cleanup
        Always call `close()` when you are done with the connection. This removes all attached listeners and closes the underlying `EventSource`.

        You can also configure an optional self-cleanup rule with `connectionClose`. This is useful for streaming APIs that emit a known terminal message such as `eos`, or a custom event such as `close`, once the stream is finished.

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection({
	url: '/api/events',
	connectionClose: { message: 'eos' },
});
```
</RenderSource>

        Or close from a custom SSE event:

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection<unknown, { close: string }>({
	url: '/api/events',
	customEvents: [{ name: 'close' }],
	connectionClose: { event: 'close' },
});
```
</RenderSource>

        Even with self-cleanup configured, it is still good practice to call `close()` directly in your normal cleanup paths. That protects you against hangs, unexpected backend behavior, or streams that never emit the expected terminal signal.

        <RenderSource language={'tsx'}>
```
const connection = new SSEConnection({ url: '/api/events' });

// later
connection.close();
```
</RenderSource>
